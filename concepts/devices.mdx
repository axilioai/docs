Every Axilio device is a real Android phone connected to a carrier network — not a simulator or emulator. When your script needs one, you **allocate** it from the Axilio fleet, interact with it through a `DeviceHandle`, and then **deallocate** it when you are done. Billing is per-second, starting the instant allocation succeeds and ending the moment the device returns to the pool, so keeping allocations short and always releasing them — even on error — is critical. The `session()` context manager handles both sides automatically, making it the safest and most idiomatic way to work with devices.

## Allocation lifecycle

<Steps>
  <Step title="Allocate a device">
    Call `client.mobile.allocate()` with your desired constraints. Axilio finds a matching device in the fleet, locks it to your account, and returns a `DeviceHandle`. Billing begins immediately.

    ```python
    handle = client.mobile.allocate(
        platform="android",
        location="us-east-1",
        apps=["com.instagram.android"],
        timeout=60.0,
    )
    ```
  </Step>
  <Step title="Use the device">
    Wrap the `DeviceHandle` with a `MobileDriver` to control the device. All driver interactions happen within this step. See [MobileDriver](/concepts/mobile-driver) for the full selector and action API.
  </Step>
  <Step title="Deallocate the device">
    Release the device so it returns to the fleet and billing stops. Call `handle.deallocate()` explicitly, or let the `session()` context manager do it automatically — including when exceptions occur.

    ```python
    handle.deallocate()
    ```
  </Step>
</Steps>

<Warning>
  If your script exits without calling `deallocate()`, the device stays locked to your account and billing continues until the allocation reaches its `expires_at` deadline. Always release devices in a `finally` block, or use `session()` instead.
</Warning>

## Using `session()` — the recommended pattern

The `session()` context manager calls `allocate()` on entry and `deallocate()` on exit, even if an exception propagates. Use it in preference to raw `allocate()` / `deallocate()` calls.

```python
from axilio.platform import Client

client = Client()

with client.mobile.session(
    platform="android",
    location="us-east-1",
    apps=["com.instagram.android"],
    timeout=60.0,
) as handle:
    # Handle is live; billing is running.
    print(handle.device_name)
    # ... drive the device ...
# Deallocated automatically here — billing stops.
```

<Tip>
  Wrap your entire script body inside a single `with client.mobile.session(...)` block. This guarantees cleanup even if `MobileDriver` raises mid-automation.
</Tip>

## The `DeviceHandle` object

`client.mobile.allocate()` and `client.mobile.session()` both return a `DeviceHandle`. Treat it as a read-only credential bundle — it carries everything the SDK needs to identify and connect to the specific device that was allocated for you.

<ResponseField name="id" type="str">
  The **allocation ID** — a unique identifier for *this specific lease*. Use this ID when calling `client.mobile.deallocate(allocation_id)` directly, and when debugging billing records.
</ResponseField>

<ResponseField name="device_id" type="str">
  The stable identifier of the physical device within the Axilio fleet. The same physical device may produce a different allocation `id` each time it is leased.
</ResponseField>

<ResponseField name="device_name" type="str">
  A human-readable label for the device, such as `"Pixel 7 Pro — rack-us-east-1-03"`. Useful for logs and debugging.
</ResponseField>

<ResponseField name="platform" type='"android" | "iphone"'>
  The operating system of the allocated device. Currently `"android"` for all general-availability allocations. `"iphone"` is available on select plans.
</ResponseField>

<ResponseField name="location" type="str">
  The rack or data-center location identifier where the device is physically hosted, for example `"us-east-1"`. Matches the `location` you requested at allocation time.
</ResponseField>

<ResponseField name="expires_at" type="datetime">
  The UTC timestamp at which Axilio will forcibly deallocate the device if it has not been released already. An idle device that stays allocated past this deadline is reclaimed automatically — plan your automation to complete before this deadline.
</ResponseField>

<ResponseField name="signaling_url" type="str">
  The WebSocket URL of the signaling service used to establish a peer connection with the device. The `MobileDriver` uses this internally; you do not typically need to read it directly.
</ResponseField>

<ResponseField name="signaling_token" type="str">
  A short-lived bearer token that authenticates the connection to the signaling service. Do not log or persist this value.
</ResponseField>

## Location pinning

Pass a `location` string to ensure your device is in a specific region. This is useful when your automation needs to interact with geo-restricted content or carrier-specific features.

```python
# Pin to a US East device
with client.mobile.session(location="us-east-1") as handle:
    print(handle.location)  # "us-east-1"
```

To discover which locations are available on your plan, call `client.mobile.locations()`:

```python
locations = client.mobile.locations()
for loc in locations:
    print(loc.id, loc.name, loc.region)
```

<Note>
  Omitting `location` lets Axilio pick the nearest available device, which usually minimises allocation wait time. Specify a location only when geographic placement matters for your use case.
</Note>

## Apps

The `apps` parameter accepts a list of bundle IDs (Android package names). Axilio routes your allocation to a device that has those apps pre-installed, saving you the setup time of installing them at runtime.

```python
with client.mobile.session(
    apps=["com.instagram.android", "com.twitter.android"]
) as handle:
    # Both apps are already installed on this device.
    ...
```

To discover which apps are available across the fleet, call `client.mobile.supported_apps()`:

```python
apps = client.mobile.supported_apps(platform="android")
for app in apps:
    print(app.bundle_id, app.name)
```

## Allocation timeout

The `timeout` parameter (default `60.0` seconds) controls how long `allocate()` waits for a matching device to become free before raising an error. Increase it if your required `apps` or `location` combination has limited availability.

```python
handle = client.mobile.allocate(
    apps=["com.example.rareapp"],
    timeout=120.0,  # wait up to 2 minutes
)
```

## Billing summary

<Info>
  Billing starts the moment `allocate()` returns a `DeviceHandle` and stops the moment `deallocate()` is acknowledged by the backend. Allocation wait time (before a device is found) is **not** billed. Usage is visible in the Axilio dashboard under **Usage → Sessions**.
</Info>

| Event | Billing effect |
|---|---|
| `allocate()` returns | Billing starts |
| `deallocate()` called | Billing stops |
| `expires_at` reached without deallocate | Billing stops (forced release) |
| Exception inside `session()` block | `deallocate()` is called automatically by the context manager |

## Manual allocate / deallocate pattern

If you cannot use the context manager (for example, when the allocation must span multiple functions), use an explicit `try / finally` block:

```python
handle = client.mobile.allocate(platform="android")
try:
    run_my_automation(handle)
finally:
    client.mobile.deallocate(handle.id)
```

`DeviceHandle` also supports the context manager protocol directly:

```python
with client.mobile.allocate(platform="android") as handle:
    run_my_automation(handle)
```
