Axilio operates in two distinct execution contexts: **local mode**, where you run scripts from your own machine using an API key, and **sandbox mode**, where your script runs inside the Axilio platform as part of a workflow run with a pre-allocated device. The `Client` detects which context it is in automatically at construction time by inspecting environment variables, so the same script code works in both environments without modification. Understanding the differences between modes helps you write scripts that behave correctly everywhere, and helps you reason about which platform operations are available to you at any given time.

## How mode detection works

When you construct `axilio.Client()`, it calls `axilio._mode.detect()` internally. Detection inspects two environment variables:

| Environment variable | Set by | Purpose |
|---|---|---|
| `AXILIO_API_KEY` | You, in your local environment | Authenticates your local SDK session |
| `AXILIO_SANDBOX_TOKEN` | Axilio platform, automatically | Short-lived token injected into each workflow run's sandbox |
| `AXILIO_ATLAS_ENDPOINT` | Axilio platform, automatically | Internal endpoint for the sandbox runtime |

If **both** `AXILIO_SANDBOX_TOKEN` and `AXILIO_ATLAS_ENDPOINT` are present, the client enters sandbox mode. Otherwise, it falls back to local mode.

```python
import axilio

client = axilio.Client()
print(client.mode)  # Mode.LOCAL or Mode.SANDBOX
```

You can compare the mode using the `Mode` enum:

```python
from axilio import Client
from axilio._mode import Mode

client = Client()

if client.mode == Mode.SANDBOX:
    print("Running inside a workflow sandbox")
elif client.mode == Mode.LOCAL:
    print("Running locally")
```

<Note>
  Never set `AXILIO_SANDBOX_TOKEN` manually in your local environment. It is only valid for the specific sandbox it was minted for, and using it outside of that context will cause authentication failures. Set `AXILIO_API_KEY` for all local usage.
</Note>

## Local mode

In local mode, the client authenticates using your `AXILIO_API_KEY`. You have full access to all SDK operations, including account management, billing, and device allocation. This is the mode you use when developing and testing automations from your laptop or CI environment.

```python
import os
import axilio

# Either set AXILIO_API_KEY in your environment, or pass it explicitly:
client = axilio.Client(api_key=os.environ["AXILIO_API_KEY"])

# Allocate a device, drive it, and clean up
with client.mobile.session(platform="android") as handle:
    print(f"Allocated: {handle.device_name}")
```

## Sandbox mode

In sandbox mode, your script is executing inside a workflow run. The Axilio platform has already allocated a device for the run before your script starts. You interact with that device through `MobileDriver.connect()`, which pairs automatically with the pre-allocated VM device over a local daemon socket — no signaling URL or token handling required.

```python
import axilio
from axilio.drivers.mobile import MobileDriver

# Client() auto-detects sandbox mode from environment variables.
client = axilio.Client()

# In sandbox: session() returns a handle to the pre-allocated device.
with client.mobile.session() as handle:
    print(f"Device: {handle.device_name} ({handle.platform})")

    with MobileDriver.connect() as driver:
        driver.wait_for_text("Home", timeout=20.0)
        driver.screenshot()
```

### Device pre-allocation in sandbox

Because the platform allocates the device before your script starts, `client.mobile.session()` in sandbox mode does not wait for a new device — it returns a handle to the already-running device immediately. The `platform`, `location`, and `apps` you specified when creating the run's workflow determine which device was allocated.

<Info>
  In sandbox mode, `deallocate()` is a no-op. The platform controls the device lifecycle, and the device is reclaimed automatically when the run reaches a terminal state.
</Info>

### `AllocationMismatchError`

If your script calls `client.mobile.allocate()` with `platform`, `location`, or `apps` arguments that do not match the device that was actually pre-allocated, Axilio raises `AllocationMismatchError` rather than silently ignoring the mismatch. This catches copy-paste errors when migrating a local script to a workflow.

```python
from axilio.errors import AllocationMismatchError

try:
    handle = client.mobile.allocate(platform="iphone")  # But sandbox has Android
except AllocationMismatchError as e:
    print(f"Mismatch: {e}")
```

<Warning>
  If you see `AllocationMismatchError` in a workflow run, check that the `platform`, `location`, and `apps` you pass to `allocate()` or `session()` in your script match the settings configured for the workflow in the dashboard.
</Warning>

## Restricted operations in sandbox mode

Sandbox mode imposes write restrictions on account-management operations. This prevents a workflow script from accidentally (or maliciously) modifying your account configuration. Read operations always succeed in both modes.

### Operations that raise `SandboxPermissionError`

Any of the following calls will raise `SandboxPermissionError` when executed in sandbox mode:

<CardGroup cols={2}>
  <Card title="Billing writes" icon="credit-card">
    - `client.billing.upgrade(...)`
    - `client.billing.downgrade(...)`
  </Card>
  <Card title="API key management" icon="key">
    - `client.api_keys.create(...)`
    - `client.api_keys.revoke(...)`
  </Card>
  <Card title="Organization writes" icon="users">
    - `client.org.invite(...)`
    - `client.org.remove_member(...)`
  </Card>
  <Card title="Everything else" icon="circle-check">
    All other operations — device allocation, driver actions, workflow/run management, and all read operations — work normally in both modes.
  </Card>
</CardGroup>

```python
from axilio.errors import SandboxPermissionError

# This will raise in sandbox mode:
try:
    client.api_keys.create(name="my-key")
except SandboxPermissionError as e:
    print(f"Not allowed in sandbox: {e}")
```

### Operations that work in both modes

The following read-only account operations work normally regardless of mode:

```python
# These all succeed in both local and sandbox mode:
balance = client.billing.balance()
subscription = client.billing.subscription()
invoices = client.billing.invoices()
usage = client.usage.metrics()
org = client.org.get()
members = client.org.members()
```

## Mode comparison summary

<Tabs>
  <Tab title="Local mode">
    **Authentication:** `AXILIO_API_KEY` environment variable or `api_key=` constructor argument.

    **Device allocation:** `client.mobile.allocate()` acquires a new device from the fleet. Billing starts immediately.

    **`MobileDriver.connect()`:** Requires an active sandbox or a transport configured with a `DeviceHandle`. Full local driving is available when the local transport is wired.

    **Account writes:** All operations permitted — billing changes, API key management, org management.

    **Typical use case:** Local development, testing, CI pipelines, one-off scripts.
  </Tab>
  <Tab title="Sandbox mode">
    **Authentication:** `AXILIO_SANDBOX_TOKEN` — injected automatically by the platform.

    **Device allocation:** `client.mobile.session()` returns a handle to the pre-allocated device. No waiting, no additional billing.

    **`MobileDriver.connect()`:** Connects immediately over the local daemon socket to the pre-allocated device.

    **Account writes:** Restricted — raises `SandboxPermissionError` for billing, API key, and org writes.

    **Typical use case:** Workflow runs dispatched from the dashboard or via `client.runs.create()`.
  </Tab>
</Tabs>

## Checking the mode at runtime

Use `client.mode` to adapt your script's behavior based on the execution context:

```python
import axilio
from axilio._mode import Mode

client = axilio.Client()

if client.mode == Mode.SANDBOX:
    # In a workflow run — device is pre-allocated.
    with client.mobile.session() as handle:
        automate(handle)
else:
    # Local — allocate a device explicitly.
    with client.mobile.session(
        platform="android",
        location="us-east-1",
        apps=["com.example.myapp"],
    ) as handle:
        automate(handle)
```
