What Axilio is, why it exists, and how the pieces fit together.

Request a device through the SDK, drive it with a chainable vision-native API, then release it — we handle provisioning, networking, and teardown.

Why it matters: emulators quietly diverge from production. Real calls don't connect, push notifications don't land on time, and carrier-gated flows behave differently than they do for a real user. Axilio closes that gap by giving you the real thing.

## Core SDK concepts

The SDK has two entry points: a `Client` that manages device lifecycle and account resources, and a `MobileDriver` that controls the device you acquire.

### `axilio.platform.Client`

Your connection to the API. Construct it once and share it — it holds your credentials and connection pool. Every resource hangs off it as an attribute.

```python
from axilio.platform import Client

client = Client()  # reads AXILIO_API_KEY from the environment
```

`client.mobile` (devices), `client.workflows` and `client.runs`, `client.usage` and `client.billing`, `client.api_keys` and `client.org`.

### `axilio.drivers.MobileDriver`

`MobileDriver` wraps the `DeviceHandle` from `allocate()` or `session()` and gives you a chainable selector surface. Every selector returns an `Element` you act on directly:

<CodeGroup>
```python Session
from axilio.drivers import MobileDriver

with client.mobile.session(platform="android") as device:
    driver = MobileDriver(device)

    driver.find_text("Sign in").tap()
    driver.find_text("Email").type_into("you@example.com")
    driver.find(query="the blue Continue button").tap()
    png = driver.screenshot()
```

```python Explicit
from axilio.drivers import MobileDriver

device = client.mobile.allocate(platform="android")
try:
    driver = MobileDriver(device)

    driver.find_text("Sign in").tap()
    driver.find_text("Email").type_into("you@example.com")
    driver.find(query="the blue Continue button").tap()
    png = driver.screenshot()
finally:
    client.mobile.deallocate(device)
```
</CodeGroup>

`find_text` uses fast on-device OCR; `find` sends a plain-English query to Argus for anything text alone can't pin down.

## Argus — vision inference

Argus is Axilio's vision service: object detection, OCR (free and premium), and a VLM that locates elements from plain-English descriptions. `driver.find(query=...)` calls it for you, or call `client.argus` directly to run inference on any image — useful for screenshot analysis or pre-computing OCR.

## Under the hood

What happens between `allocate()` and your first `tap()` — it explains the API's shape and where latency comes from.

1. **Allocation.** When you call `client.mobile.allocate()`, the platform finds a free physical phone matching your constraints (platform, location, pre-installed apps), locks it to your account, and returns a [`DeviceHandle`](/concepts/devices) carrying a signaling URL and a short-lived token. Billing starts here.
2. **Connection.** `MobileDriver` uses the handle to open a peer connection to the device over the signaling service. Input (taps, swipes, text, key presses) travels down this channel; frames travel back up. There's no emulator in the path — these are real touch events on real glass.
3. **On-device agent.** A small agent running alongside the phone executes your instructions against the hardware and reports structured results (or [errors](/reference/errors)) back over the wire. This is why driver errors carry a `.retryable` flag — the agent knows whether a failure was transient.
4. **Vision.** When you call `find(query=...)`, the current frame is sent to [Argus](/api-reference/argus/overview), Axilio's inference service, which returns element coordinates. `find_text()` runs OCR on the same frames. Vision is a separate service so it can scale independently of the device fleet.
5. **Teardown.** `deallocate()` releases the lease, the phone returns to the pool, and billing stops. The `session()` context manager does this for you even when your code raises.

<Info>
  Because input and frames stream over a live connection, a device stays "warm" for the whole session — there's no per-action cold start. Keep allocations short and do all your work inside one `session()` block.
</Info>
