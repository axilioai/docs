Instead of querying one label at a time, `observe()` captures **everything on
screen at once** — all recognized text and icons — as a single `Screen` you can
inspect and query locally.

```python
from axilio.platform import Client

client = Client()

with client.session("ANDROID") as driver:
    screen = driver.observe()
    print(len(screen.texts), "text elements,", len(screen.icons), "icons")
```

## The Screen object

`observe()` returns a `Screen`:

| Field | Description |
|---|---|
| `texts` | A list of [`Element`](/driver/actions)s — every recognized text item. |
| `icons` | A list of detected icons, each with a `bbox`, `center`, and `confidence`. |
| `width`, `height` | The screen dimensions, in pixels. |
| `captured_at` | When the snapshot was taken. |
| `hash` | A content hash, handy for detecting whether the screen changed. |

## Query without re-capturing

A `Screen` is a snapshot. Querying it with `find_text` / `find_all_text` searches
the captured data **without going back to the device** — fast, and consistent
across every query against that snapshot:

```python
screen = driver.observe()

sign_in = screen.find_text("Sign in")     # Element | None
rows = screen.find_all_text(contains="$")  # list[Element]
```

This is the difference between `screen.find_text(...)` and the driver's own
[`driver.find_text(...)`](/driver/find-by-text): the driver captures a fresh screen
on every call, while `screen.find_text(...)` reuses the one snapshot. Use `observe`
when you want to make several decisions about a single, stable view.

<Tip>
  Compare `screen.hash` across two `observe()` calls to tell whether the UI changed
  — a lightweight alternative to diffing screenshots.
</Tip>

## Observe or screenshot?

- **`observe()`** gives you structured, queryable text and icons — use it to *make
  decisions* in code.
- **[`screenshot()`](/driver/screenshots)** gives you raw PNG bytes — use it for
  logging, audit trails, or handing pixels to your own model.
