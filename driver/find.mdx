`find` locates something on the screen so you can act on it. There are two ways:
by its **text**, or by **describing** it.

## By its text

`find_text` finds anything with a visible label, like a button or a menu item.
Fast and exact.

```python
driver.find_text("Settings").tap()
```

It hands back what it found, or nothing if it's not there:

```python
result = driver.find_text("Sign in")
if result:
    result.tap()
```

<Warning>
  Because it can come back empty, calling `.tap()` right away can fail if the
  screen hasn't loaded. To be safe, [wait for the text](/driver/wait). That waits
  until it appears, then taps it:

  ```python
  driver.wait_for_text("Sign in").tap()
  ```
</Warning>

Get every match with `find_all_text`. Handy for lists and feeds:

```python
for item in driver.find_all_text(contains="Notification"):
    print(item.text)
```

## By describing it

When there's no text to match, like an icon or an image, describe it in plain
English with `find`:

```python
driver.find(query="the heart icon under the photo").tap()
```

Unlike `find_text`, `find` speaks up when it can't find what you asked for:

```python
from axilio.drivers.mobile import ElementNotFoundError

try:
    driver.find(query="the shopping cart").tap()
except ElementNotFoundError:
    print("no cart on this screen")
```

<Note>
  Describing something uses a vision model, so it costs a little more than finding
  by text. On steps you run a lot, prefer text. You can
  [see the per-call cost](/observability/cost) of every vision call.
</Note>

## Which to use

- **Text** when there's a label you can read. It's faster and exact.
- **Describing** when there isn't: an icon, or something easier to describe than
  to name.
