Every request the Axilio SDK makes is authenticated with an API key — a long-lived secret that identifies your organisation and controls access to all platform resources. Keys take the form `ax_live_…` and are sent to the API in the `X-Axilio-Api-Key` request header. The SDK handles header injection automatically; you simply supply the key once at startup and forget about it. This guide explains how to create a key, how to pass it to the client, how to manage keys over time, and what happens when authentication fails.

## Create an API key

You can create a key either through the dashboard UI or programmatically using the SDK. Both paths produce an `APIKeyCreated` object whose `secret` field holds the only copy of the plaintext you will ever see.

### Via the dashboard

<Steps>
  <Step title="Open API key settings">
    Sign in at [axilio.ai](https://axilio.ai) and navigate to **Settings → API Keys**.
  </Step>
  <Step title="Create a new key">
    Click **New key**, enter a descriptive name (e.g. `production-worker` or `local-dev`), and confirm.
  </Step>
  <Step title="Copy the secret">
    Copy the full key value from the confirmation dialog immediately. The dashboard will never show the plaintext again — only the key's name and an 8-character prefix for identification.
  </Step>
</Steps>

### Via the SDK

If you already have an active key, you can mint additional keys programmatically. This is useful for automated provisioning scripts or CI pipelines that need their own isolated credentials.

```python
from axilio.platform import Client

client = Client()  # authenticates with your existing AXILIO_API_KEY

new_key = client.api_keys.create(name="ci-pipeline")
print(new_key.secret)   # store this now — it won't be shown again
print(new_key.id)       # use this id later if you need to revoke the key
print(new_key.prefix)   # first 8 chars, safe to log for identification
```

<Warning>
  The `secret` field on `APIKeyCreated` is the **only time** the plaintext is accessible. The Axilio backend stores only a hash; neither the dashboard nor `client.api_keys.list()` will return the full value again. Save the secret to a secrets manager or environment variable before discarding the object.
</Warning>

## Pass the key to the SDK

The SDK resolves your API key in this order of precedence:

1. The `api_key` constructor argument.
2. The `AXILIO_API_KEY` environment variable.
3. (Inside sandbox workflow runs) the `AXILIO_SANDBOX_TOKEN` environment variable, injected automatically by the runtime.

### Environment variable (recommended)

Set `AXILIO_API_KEY` in your shell or deployment environment and construct the client with no arguments. This keeps secrets out of source code entirely.

```bash
export AXILIO_API_KEY="ax_live_••••••••••••••••"
```

```python
from axilio.platform import Client

client = Client()  # key resolved from environment — nothing hard-coded
```

### Constructor argument

Pass the key directly when you need programmatic control — for example, when reading from a secrets manager at runtime.

```python
import boto3
from axilio.platform import Client

secret = boto3.client("secretsmanager").get_secret_value(SecretId="axilio/api-key")["SecretString"]
client = Client(api_key=secret)
```

### Raw HTTP header

If you are calling the Axilio REST API directly — without the Python SDK — include the key in every request as a custom header:

```http
GET /v1/mobile/available HTTP/1.1
Host: api.axilio.ai
X-Axilio-Api-Key: ax_live_••••••••••••••••
```

<CodeGroup>
```bash cURL
curl https://api.axilio.ai/v1/mobile/available \
  -H "X-Axilio-Api-Key: $AXILIO_API_KEY"
```

```python httpx
import httpx

resp = httpx.get(
    "https://api.axilio.ai/v1/mobile/available",
    headers={"X-Axilio-Api-Key": "ax_live_••••••••••••••••"},
)
```
</CodeGroup>

## Manage your keys

### List active keys

Retrieve all active API keys for your organisation. The list response never includes the plaintext secret — only the key metadata and the 8-character prefix.

```python
keys = client.api_keys.list()

for key in keys:
    print(key.id, key.name, key.prefix, key.created_at, key.last_used_at)
```

The fields returned for each key:

<ResponseField name="id" type="string" required>
  Unique identifier for the key. Use this with `revoke()`.
</ResponseField>

<ResponseField name="name" type="string" required>
  Human-readable label you set at creation time.
</ResponseField>

<ResponseField name="prefix" type="string" required>
  First 8 characters of the key — safe to log or display for identification without exposing the secret.
</ResponseField>

<ResponseField name="created_at" type="datetime" required>
  UTC timestamp of key creation.
</ResponseField>

<ResponseField name="last_used_at" type="datetime | None" required>
  UTC timestamp of the most recent successful authentication, or `None` if the key has never been used.
</ResponseField>

### Revoke a key

Permanently revoke a key by its `id`. The key stops authenticating immediately; any in-flight requests that already carry it will complete normally, but no new requests will succeed.

```python
client.api_keys.revoke(key_id="apk_xxxxxxxxxxxxxxxx")
```

<Note>
  API key management (`list`, `create`, `revoke`) is available in **local mode only** — that is, when you run code on your own machine or infrastructure. Inside a sandboxed workflow run, these operations raise `SandboxPermissionError`. This restriction exists because a running automation script should not be able to mint or destroy the credentials that govern it.
</Note>

## `APIKeyCreated` reference

<ResponseField name="id" type="string" required>
  Unique key identifier. Persist this if you need to revoke the key later.
</ResponseField>

<ResponseField name="name" type="string" required>
  The label you passed to `create(name=...)`.
</ResponseField>

<ResponseField name="secret" type="string" required>
  The full plaintext API key. Visible exactly once at creation — the backend stores only a hash.
</ResponseField>

<ResponseField name="prefix" type="string" required>
  First 8 characters of the secret, for display and logging.
</ResponseField>

<ResponseField name="created_at" type="datetime" required>
  UTC timestamp when the key was minted.
</ResponseField>

## Handle authentication errors

When a request reaches the Axilio API with a missing, malformed, or revoked key, the backend returns HTTP `401` and the SDK raises `UnauthorizedError`.

```python
from axilio.platform import Client, UnauthorizedError

try:
    client = Client(api_key="ax_live_invalid")
    devices = client.mobile.available()
except UnauthorizedError as exc:
    print(f"Authentication failed ({exc.status_code}): {exc.message}")
    # Re-check AXILIO_API_KEY, verify the key hasn't been revoked,
    # and confirm you're using the right key for this organisation.
```

<Tip>
  If you receive an `UnauthorizedError` unexpectedly, verify that: the key hasn't been revoked via the dashboard or `revoke()`; you're not accidentally loading an old key from a `.env` file that overrides the intended value; and the key belongs to the organisation you expect (each key is scoped to exactly one org).
</Tip>

## Security best practices

Keep your API keys secure with these habits:

- **Never commit keys to version control.** Add `.env` to your `.gitignore` and use a secrets manager (AWS Secrets Manager, HashiCorp Vault, GitHub Actions secrets, etc.) in CI/CD pipelines.
- **Use one key per service or environment.** Give each application, pipeline, or developer their own key. This limits blast radius if a key is leaked and makes audit logs meaningful — `last_used_at` tells you exactly which service was active.
- **Rotate keys regularly.** Create a new key, update the consuming service, verify it works, then revoke the old key. The `prefix` field lets you confirm which key a service is using without exposing the secret.
- **Set a short `timeout` in untrusted environments.** The `Client` constructor accepts a `timeout` parameter (default `30.0` seconds). Lower it in latency-sensitive contexts to avoid hanging on network faults.
- **Revoke keys immediately on suspected compromise.** Revocation takes effect instantly — there's no delay between calling `revoke()` and the key ceasing to work.
