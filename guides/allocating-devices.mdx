Every interaction with a mobile device on Axilio starts with an allocation — a lease that reserves one physical device for your exclusive use. Axilio gives you two paths to manage that lease: the `session()` context manager, which handles allocation and release automatically, and the lower-level `allocate()` / `deallocate()` pair for cases where you need finer-grained control. This guide walks through both approaches, explains how to filter and discover devices before you commit, and covers the error handling you need to build robust automation pipelines.

## The session context manager (recommended)

The `session()` context manager is the safest and most concise way to work with a device. It allocates a device when the `with` block is entered and guarantees `deallocate()` is called when the block exits — even if an exception is raised inside it. Use `session()` for all production code unless you have a specific reason to manage the device lifecycle manually.

```python
from axilio.platform import Client
from axilio.drivers import MobileDriver

client = Client()  # reads AXILIO_API_KEY from the environment

with client.mobile.session(platform="android") as device:
    driver = MobileDriver(device)
    driver.find_text("Sign in").tap()
    png = driver.screenshot()
# device is automatically released here, even if an exception occurred
```

`session()` accepts the same filtering arguments as `allocate()`:

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `platform` | `"android"` \| `"iphone"` | `"android"` | Target device OS. |
| `location` | `str \| None` | `None` | Pin to a specific data-center location ID. |
| `apps` | `list[str] \| None` | `None` | Require these app bundle IDs to be pre-installed. |
| `timeout` | `float` | `60.0` | Seconds to wait for a free device before raising. |

<Warning>
Never call `device.deallocate()` yourself inside a `with client.mobile.session()` block. The context manager's `__exit__` handler calls it for you; a double-deallocate will raise an error and may interrupt other allocations in your account.
</Warning>

## Checking availability before allocating

Before committing to an allocation, call `client.mobile.available()` to see how many devices currently match your requirements. This is useful in batch jobs where you want to back off gracefully rather than block on the `timeout` parameter.

```python
result = client.mobile.available(
    platform="android",
    location="us-east-1",
    apps=["com.example.myapp"],
)

print(f"{len(result.devices)} device(s) free out of {result.total_in_fleet} in fleet")

for device in result.devices:
    print(f"  {device.name} — {device.model}, Android {device.os_version}")
```

`available()` returns an `AvailableDevices` object with two fields:

- **`devices`** — a list of `AvailableDevice` records (each has `id`, `name`, `platform`, `location`, `model`, `os_version`) that are free *right now* and match your filter.
- **`total_in_fleet`** — the total number of devices matching your filter, regardless of current availability. Use this to understand capacity headroom.

<Note>
Availability is a snapshot in time. A device that appears free in `available()` may be taken by another caller before your `allocate()` arrives. Always handle the allocation timeout described below.
</Note>

## Listing available locations

Axilio runs device racks in multiple data-center regions. Call `client.mobile.locations()` to see the full list of location IDs you can pass to `session()` or `allocate()`.

```python
locations = client.mobile.locations()

for loc in locations:
    print(f"{loc.id:20s} {loc.name} ({loc.region})")
```

Each `Location` has three fields: `id` (the string you pass to `allocate()`), `name` (a human-readable label), and `region` (the broad geographic region, e.g. `"us-east"` or `"eu-west"`).

**When to pin a location:**

- Your test data, backend, or CDN is hosted in a specific region and you want minimal network latency between the device and the service under test.
- Compliance requirements restrict data to a particular geography.
- You're comparing behavior across regions (run the same workflow in `"us-east-1"` and `"eu-west-1"` in parallel).

If you omit `location`, Axilio picks the region with the most available capacity, which generally minimises queue time.

## Filtering by pre-installed apps

Pass a list of app bundle IDs to `apps` to request a device that already has those apps installed. Axilio will only allocate devices from the pool that has your required apps.

```python
with client.mobile.session(
    platform="android",
    apps=["com.example.myapp", "com.google.android.gms"],
) as device:
    driver = MobileDriver(device)
    # com.example.myapp is guaranteed to be installed
    driver.find_text("Launch App").tap()
```

To discover which apps are available on the fleet, call `client.mobile.supported_apps()`:

```python
# All apps on Android
apps = client.mobile.supported_apps(platform="android")

# Filter by category
social_apps = client.mobile.supported_apps(platform="android", category="social")

for app in social_apps:
    print(f"{app.bundle_id:40s} {app.name}")
```

Each `App` has `bundle_id`, `name`, `platform`, and `category` (may be `None`).

## Manual allocate / deallocate

For advanced use cases — such as pre-warming a pool of devices or handing a `DeviceHandle` across function boundaries — use `allocate()` and `deallocate()` directly. The signature is identical to `session()`.

```python
from axilio.platform import Client, AxilioError
from axilio.drivers import MobileDriver

client = Client()

device = client.mobile.allocate(
    platform="android",
    location="us-east-1",
    timeout=30.0,
)

try:
    driver = MobileDriver(device)
    driver.find_text("Home").tap()
finally:
    # Always release in a finally block to avoid leaking allocations.
    client.mobile.deallocate(device.id)
```

`DeviceHandle` also implements the context manager protocol, so you can use it with `with` directly if you've already called `allocate()`:

```python
device = client.mobile.allocate(platform="iphone")

with device:
    driver = MobileDriver(device)
    driver.find_text("Settings").tap()
# deallocate() is called automatically on __exit__
```

## Understanding DeviceHandle

`allocate()` (and `session()`) returns a `DeviceHandle` — a frozen dataclass that describes the acquired device and its lease.

| Field | Type | Description |
|-------|------|-------------|
| `id` | `str` | The allocation ID. Pass to `deallocate()` for manual release. |
| `device_id` | `str` | The physical device's stable identifier. |
| `device_name` | `str` | Human-readable device label, e.g. `"Pixel 7a – rack-3"`. |
| `platform` | `"android"` \| `"iphone"` | The OS of the allocated device. |
| `location` | `str` | The location ID where the device resides. |
| `expires_at` | `datetime` | UTC timestamp when the lease expires automatically. |
| `signaling_url` | `str` | WebRTC signaling endpoint for the device connection. |
| `signaling_token` | `str` | Short-lived credential used to authenticate the signaling session. |

<Info>
`expires_at` is a hard deadline set by the Axilio backend. If your automation runs longer than expected and the lease expires, the device will be reclaimed even if you haven't called `deallocate()`. Monitor `expires_at` in long-running jobs and re-allocate a fresh device before the deadline if needed.
</Info>

## Handling allocation timeout

When no device matching your filter is free, `allocate()` (and `session()`) polls the server until `timeout` seconds have elapsed, then raises `axilio.AxilioError`. Catch it to implement a retry or fallback strategy.

```python
import time
from axilio.platform import Client, AxilioError, RateLimitError
from axilio.drivers import MobileDriver
from axilio.drivers.mobile import TimeoutError as DeviceTimeoutError

client = Client()

MAX_ATTEMPTS = 3

for attempt in range(1, MAX_ATTEMPTS + 1):
    try:
        with client.mobile.session(
            platform="android",
            apps=["com.example.myapp"],
            timeout=60.0,
        ) as device:
            driver = MobileDriver(device)
            driver.find_text("Home").tap()
            break  # success — exit the retry loop

    except RateLimitError:
        wait = 2 ** attempt
        print(f"Rate limited. Retrying in {wait}s…")
        time.sleep(wait)

    except AxilioError as exc:
        print(f"Attempt {attempt} failed: {exc}")
        if attempt == MAX_ATTEMPTS:
            raise
        time.sleep(5)
```

<Tip>
Pass a longer `timeout` (e.g. `120.0`) during peak hours rather than retrying immediately — the backend queues your allocation request and dispatches a device as soon as one becomes free, so a longer timeout is more efficient than repeated short attempts.
</Tip>

## Complete example with error handling

The following example combines availability checking, location selection, app filtering, and robust error handling into a single production-ready pattern.

```python
import time
from axilio.platform import Client, AxilioError, RateLimitError
from axilio.drivers import MobileDriver

client = Client()

REQUIRED_APPS = ["com.example.myapp"]
PLATFORM = "android"


def run_automation() -> bytes:
    """Allocate a device, run a task, and return a screenshot."""

    # 1. Check capacity before blocking on an allocation.
    available = client.mobile.available(platform=PLATFORM, apps=REQUIRED_APPS)
    if available.total_in_fleet == 0:
        raise RuntimeError(
            f"No {PLATFORM} devices with {REQUIRED_APPS} exist in the fleet."
        )
    if not available.devices:
        print("All matching devices are busy — will wait for one to free up…")

    # 2. Acquire the device; session() guarantees release.
    with client.mobile.session(
        platform=PLATFORM,
        apps=REQUIRED_APPS,
        timeout=90.0,
    ) as device:
        print(f"Allocated {device.device_name} in {device.location}")
        print(f"Lease expires at {device.expires_at.isoformat()}")

        driver = MobileDriver(device)

        # 3. Navigate to the feature under test.
        driver.wait_for_text("Home", timeout=15.0)
        driver.find_text("Settings").tap()
        driver.wait_for_text("Account", timeout=10.0)

        # 4. Capture a screenshot for verification.
        return driver.screenshot()


if __name__ == "__main__":
    png_bytes = run_automation()
    with open("result.png", "wb") as f:
        f.write(png_bytes)
    print("Screenshot saved to result.png")
```
