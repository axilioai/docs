Building automation that works reliably in production requires more than a script that passes on the first try. Real devices have transient failures, apps have loading states, and long-running pipelines surface edge cases that smoke tests miss. The ten practices below are distilled from production Axilio deployments — follow them to write automation that is fast, debuggable, and easy to maintain.

---

## 1. Always use `with client.mobile.session()`

The `session()` context manager is the only way to guarantee that a device is released even when your code raises an exception. Manual `allocate()` / `deallocate()` patterns almost always result in leaked allocations when something goes wrong mid-script — and leaked allocations consume billing quota until their lease expires.

<CodeGroup>

```python Recommended
with client.mobile.session(platform="android") as device:
    driver = MobileDriver(device)
    driver.find_text("Sign in").tap()
# Released automatically — even on exception
```

```python Avoid
device = client.mobile.allocate(platform="android")
driver = MobileDriver(device)
driver.find_text("Sign in").tap()
# If the line above raises, deallocate() is never called
client.mobile.deallocate(device.id)
```

</CodeGroup>

If you genuinely need to manage the lifecycle manually, use `with device:` — `DeviceHandle` also implements the context manager protocol.

## 2. Prefer `find_text` for speed

`find_text()` runs on-device OCR and returns results in milliseconds. `find()` with a natural-language query makes a round-trip to Axilio's VLM infrastructure and is 5–20× slower. Use `find_text()` as your default; reach for `find()` only when the element has no readable text.

<CodeGroup>

```python Recommended — fast
driver.find_text("Continue").tap()
driver.find_text("Email").type_into("you@example.com")
```

```python Use sparingly — slower
# Only when "Continue" is an icon with no text label
driver.find(query="the blue forward arrow button at the bottom").tap()
```

</CodeGroup>

## 3. Use `observe()` to batch queries

Every `find_text()` call on the driver issues a fresh `observe()` internally — one OCR round-trip per call. If you need to check several elements at once, call `observe()` once and query the returned `Screen` object multiple times at no extra cost.

<CodeGroup>

```python Recommended — one OCR call
screen = driver.observe()
username_field = screen.find_text("Username")
password_field = screen.find_text("Password")
error_banner   = screen.find_text("Invalid credentials")

if error_banner:
    raise AssertionError(f"Login error: {error_banner.text}")
username_field.type_into("alice")
password_field.type_into("s3cr3t")
```

```python Avoid — three separate OCR calls
username_field = driver.find_text("Username")   # 1st OCR
password_field = driver.find_text("Password")   # 2nd OCR
error_banner   = driver.find_text("Invalid")    # 3rd OCR
```

</CodeGroup>

## 4. Set appropriate timeouts on every wait call

The default `timeout=10.0` on `find()` and `wait_*` methods is a safe starting point for development, but production flows vary. A slow network call might take 25 seconds; a quick transition might only need 3. Calibrate timeouts to your app's real behaviour — too short causes flaky failures, too long wastes time when something is genuinely broken.

```python
# Short timeout for transitions that should be instant.
driver.wait_until_gone("Submitting…", timeout=3.0)

# Generous timeout for server round-trips.
driver.wait_for_text("Order confirmed", timeout=30.0)

# Very generous timeout for cold-start / first-launch scenarios.
driver.wait_until_gone("Loading", timeout=60.0, poll_ms=500)
```

<Tip>
Set `poll_ms` to a larger value (500–1000 ms) for long waits to reduce OCR frequency and lower costs. Use the default 300 ms only for fast-changing UI where you need tight loop timing.
</Tip>

## 5. Handle `ElementNotFoundError` gracefully

`driver.find()` always raises `ElementNotFoundError` when the element is absent. `driver.find_text()` returns `None`. Never call `.tap()` or other actions on an unchecked `find_text()` result — Python will raise `AttributeError` on `None` at runtime, which is harder to diagnose than a proper error.

<CodeGroup>

```python Recommended
from axilio.drivers.mobile import ElementNotFoundError

# find_text returns None — always check.
banner = driver.find_text("Promotional offer")
if banner is not None:
    banner.tap()

# find raises — catch it when the element is optional.
try:
    driver.find(query="the dismiss X on the survey popup").tap()
except ElementNotFoundError:
    pass  # no popup — that's fine
```

```python Avoid
# AttributeError: 'NoneType' object has no attribute 'tap'
driver.find_text("Promotional offer").tap()
```

</CodeGroup>

## 6. Use `wait_for_text` / `wait_until_gone` instead of `time.sleep()`

`time.sleep()` pauses for a fixed duration regardless of whether the UI has settled. If the app responds faster than expected, you waste time; if it responds slower, you get a flaky failure. The `wait_*` methods poll at a configurable interval and return as soon as the condition is met.

<CodeGroup>

```python Recommended — proceeds as soon as the UI is ready
driver.find_text("Submit").tap()
driver.wait_until_gone("Processing…", timeout=20.0)
driver.wait_for_text("Confirmation", timeout=10.0)
```

```python Avoid — brittle fixed sleep
import time
driver.find_text("Submit").tap()
time.sleep(5)  # Might be too short or wastefully long
driver.find_text("Confirmation").tap()
```

</CodeGroup>

## 7. Store screenshots on failure

When automation fails, the most valuable debugging artifact is a screenshot of what the device was showing at the moment of failure. Wrap your main script body in a `try/except` block and call `driver.screenshot()` in the exception handler.

```python
import time
from pathlib import Path
from axilio.drivers.mobile import MobileDriver

driver = MobileDriver.connect()

try:
    driver.wait_for_text("Home", timeout=15.0)
    driver.find_text("Settings").tap()
    # ... rest of the flow ...

except Exception as exc:
    # Save a diagnostic screenshot before re-raising.
    timestamp = int(time.time())
    png = driver.screenshot()
    Path(f"failure_{timestamp}.png").write_bytes(png)
    print(f"Failure screenshot saved. Error: {exc}")
    raise
```

In workflow scripts, write the PNG to `/tmp/` or log it as base64 so it appears in the run's output logs.

## 8. Test workflow scripts in sandbox mode

Before deploying to a production device pool, use the Axilio sandbox to validate your workflow logic. In the dashboard, toggle **Sandbox mode** on the workflow's settings page. Sandbox runs execute on a pre-provisioned virtual device and don't count toward your live device quota — they're ideal for rapid iteration.

<Steps>
  <Step title="Open the workflow">
    Go to **Workflows**, select your workflow, and click **Edit**.
  </Step>
  <Step title="Enable sandbox mode">
    Toggle **Run in sandbox** under the **Execution** panel on the right.
  </Step>
  <Step title="Trigger a test run">
    Click **Run now** and pass representative test variables. The run appears in the **Runs** tab with a sandbox badge.
  </Step>
  <Step title="Inspect the output">
    Click the run to see logs, `print()` output, and any screenshots. Fix issues and iterate before switching to live devices.
  </Step>
</Steps>

## 9. Rotate API keys regularly

API keys grant full access to your Axilio organisation — treat them like passwords. Rotate keys on a schedule, scope them to the minimum necessary capabilities, and never embed them directly in source code.

<Warning>
Never commit an API key to a source-code repository, even a private one. Use environment variables or a secrets manager in all environments.
</Warning>

```python
# Correct — read from the environment at runtime.
import os
from axilio.platform import Client

client = Client(api_key=os.environ["AXILIO_API_KEY"])
```

```python
# Never do this — hard-coded keys end up in version history.
client = Client(api_key="ax_live_abc123...")
```

To rotate a key, go to **Settings → API Keys** in the dashboard, create a new key, update your secrets store, and then revoke the old key via the dashboard or with `client.api_keys.revoke(key_id)`.

## 10. Monitor usage to avoid billing surprises

Parallel sessions accumulate cost quickly. Regularly check your active session count and spend rate with `client.usage.metrics()` so you catch runaway loops or misconfigured batch jobs before they become expensive.

```python
metrics = client.usage.metrics()
print(f"Active sessions:  {metrics.active_sessions}")
print(f"Sessions today:   {metrics.sessions_today}")
print(f"Spend today:      ${metrics.spend_today_usd:.4f}")
print(f"Spend this month: ${metrics.spend_month_usd:.2f}")
```

Set up a billing alert in the dashboard (**Billing → Alerts**) to receive an email or webhook notification when your spend exceeds a threshold you define. This is your safety net for unexpected usage spikes.

## Quick-reference checklist

<Accordion title="Pre-deployment checklist">

Use this checklist before pushing a workflow to production:

- [ ] All `session()` calls use the context manager (`with client.mobile.session()`).
- [ ] `find_text()` return values are checked for `None` before acting.
- [ ] `ElementNotFoundError` is caught around `find()` calls on optional elements.
- [ ] Every `wait_*` call has a calibrated `timeout` — no defaults left in production paths.
- [ ] `time.sleep()` is replaced with `wait_for_text()` or `wait_until_gone()` for UI transitions.
- [ ] Failure paths call `driver.screenshot()` and persist the result.
- [ ] API key is read from an environment variable, not hard-coded.
- [ ] The workflow has been tested in sandbox mode.
- [ ] `observe()` is used when checking multiple elements on the same screen.
- [ ] Parallel worker count is within your plan's concurrency limit.

</Accordion>
