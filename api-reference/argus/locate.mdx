The `/locate` endpoint finds a UI element described in plain English — *"the login button"*, *"the search field"*, *"the accept all cookies link"* — by passing your image and query to a vision-language model (VLM). The model returns either a free-form pixel bounding box it drew itself, or an index into a list of OCR text elements you pre-computed and supplied as grounding context. Because the model does not re-run OCR internally, the recommended pattern is to call `/infer` first, cache the OCR results keyed by the frame's perceptual hash, and then pass those results as `texts` on every `/locate` call — you pay for OCR once per unique frame, not once per query.

## The Two-Step Flow

```
┌─────────────────────────┐        ┌──────────────────────────────────┐
│  POST /infer             │        │  POST /locate                     │
│  inference_type=combined │──────▶ │  query="the login button"         │
│                          │        │  texts=[...ocr_results from infer]│
│  returns:                │        │                                   │
│  • ocr_results  ─────────┼───────▶│  (VLM uses texts as grounding)    │
│  • icon_detections       │        │                                   │
│  • hash_result (cache key)│        │  returns:                         │
└─────────────────────────┘        │  • found, bbox, matched_text_index│
                                    └──────────────────────────────────┘
```

Run `/infer` once per unique frame. Store the `ocr_results` and the `hash_result.phash` as a cache key. On every element-location query, look up the cached OCR and pass it to `/locate`. When the frame changes (new `phash`), run `/infer` again.

## Request

**`POST https://api.axilio.ai/api/v1/inference/locate`**

**Authentication:** `X-Axilio-Api-Key` header required.

### Parameters

<ParamField body="image" type="string" required>
  Base64-encoded image (PNG or JPEG). Do not include a data-URI prefix.
</ParamField>

<ParamField body="query" type="string" required>
  Natural-language description of the element you want to find. Must be at least one character. Write queries that describe what you see, not XPath or CSS — for example, `"the blue submit button"` or `"the user email input field"`.
</ParamField>

<ParamField body="texts" type="TextElementInput[]">
  Pre-computed OCR elements to give the VLM grounding context. When you supply this list, the model can ground its answer to a known text region on screen and return a `matched_text_index` instead of estimating a free-form box. Omitting `texts` (or passing an empty array) tells Argus to skip OCR grounding — the VLM will locate the element from the image alone.

  Each `TextElementInput` has:
  - `text` — the recognised string from `/infer`'s `ocr_results`
  - `bbox` — the normalised [0, 1] bounding box in the source image
</ParamField>

<ParamField body="model" type="string | null">
  VLM model identifier to use for this call. Must be one of the IDs returned by `GET /api/v1/inference/models`. Omit (or pass `null`) to use the server's current default model. Use `/models` to compare context windows and per-token pricing before choosing a non-default model.
</ParamField>

## Response

<ResponseField name="found" type="boolean">
  `true` if the VLM located the element described by `query`; `false` if it determined the element is not present in the image. When `false`, `bbox` and `matched_text_index` are both `null`.
</ResponseField>

<ResponseField name="confidence" type="float">
  The model's self-reported confidence in its answer, in [0, 1]. This is a heuristic from the model — treat it as a signal, not a guarantee.
</ResponseField>

<ResponseField name="bbox" type="LocateBBox | null">
  Pixel-space bounding box drawn by the VLM when the answer is not grounded to an OCR text element. Uses a top-left origin with `x`, `y`, `width`, `height` — note this differs from the `x1, y1, x2, y2` schema used in `/infer`.

  `null` when `matched_text_index` is set or `found` is `false`.

  **HTTP response fields:**
  <Expandable title="LocateBBox fields">
    <ResponseField name="x" type="integer">Top-left x coordinate, in pixels (≥ 0).</ResponseField>
    <ResponseField name="y" type="integer">Top-left y coordinate, in pixels (≥ 0).</ResponseField>
    <ResponseField name="width" type="integer">Width of the box in pixels (> 0).</ResponseField>
    <ResponseField name="height" type="integer">Height of the box in pixels (> 0).</ResponseField>
  </Expandable>

  **Python SDK:** The `bbox` field on the `LocateResult` object is a `BoundingBox` with `x1, y1, x2, y2` pixel coordinates, consistent with `/infer` detections.
</ResponseField>

<ResponseField name="matched_text_index" type="integer | null">
  When the VLM grounded its answer to one of the `texts` you provided, this field contains the zero-based index into that list. Use it to look up the original `OcrResult` from your `/infer` response for the pixel-accurate bounding box. `null` when the answer uses a free-form `bbox` or `found` is `false`.
</ResponseField>

<ResponseField name="model" type="string">
  The model identifier actually used for this call, as reported by the provider. Useful for logging when you omit the `model` field and let the server choose.
</ResponseField>

<ResponseField name="cost_microdollars" type="integer">
  Provider-reported cost for this VLM call, in microdollars (millionths of a USD). Divide by `1_000_000` to get USD. Use this field to attribute cost per query in your own billing or observability stack.
</ResponseField>

<ResponseField name="latency_ms" type="integer">
  End-to-end VLM call latency in milliseconds, as measured by Argus. Excludes network time from your client to the Argus API.
</ResponseField>

<ResponseField name="prompt_tokens" type="integer">
  Input tokens billed for this call. Corresponds to the image and grounding context.
</ResponseField>

<ResponseField name="completion_tokens" type="integer">
  Output tokens billed for this call. Typically small — the model returns a compact structured answer.
</ResponseField>

## Examples

<CodeGroup>

```python SDK — with OCR grounding
import axilio
from axilio.argus import TextElement

client = axilio.Client()

with open("screenshot.png", "rb") as f:
    image_bytes = f.read()

# Step 1: analyse the frame
infer_result = client.argus.infer(image_bytes, inference_type="combined")

# Step 2: build grounding context from OCR output
# TextElement.bbox must be normalised to [0, 1]; OcrResult.bbox is in pixels.
from axilio.argus import BoundingBox
w = infer_result.image_width
h = infer_result.image_height
texts = [
    TextElement(
        text=r.text,
        bbox=BoundingBox(
            x1=r.bbox.x1 / w,
            y1=r.bbox.y1 / h,
            x2=r.bbox.x2 / w,
            y2=r.bbox.y2 / h,
        ),
    )
    for r in infer_result.ocr_results
]

# Step 3: locate the element
location = client.argus.locate(
    image_bytes,
    query="the login button",
    texts=texts,
)

if location.found:
    if location.matched_text_index is not None:
        # Grounded answer — get pixel bbox from original OCR result
        ocr = infer_result.ocr_results[location.matched_text_index]
        print(f"Matched text: {ocr.text!r}  bbox={ocr.bbox}")
    else:
        # Free-form VLM bbox
        b = location.bbox
        print(f"VLM bbox: x1={b.x1} y1={b.y1} x2={b.x2} y2={b.y2}")
else:
    print("Element not found on screen")
```

```python SDK — without grounding
# Omit texts to let the VLM reason from the image alone.
# Useful when you haven't pre-run /infer, but less accurate
# on text-heavy interfaces.
location = client.argus.locate(
    image_bytes,
    query="the avatar icon in the top-right corner",
)
```

```python SDK — choose a specific model
location = client.argus.locate(
    image_bytes,
    query="the accept cookies button",
    texts=texts,
    model="anthropic/claude-sonnet-4-5",
)
if location.found:
    print(f"Element found  confidence={location.confidence:.2f}")
```

## bbox vs. matched_text_index

When the VLM locates an element, it returns its answer in exactly one of two forms:

| Field | When it's set | How to use it |
|---|---|---|
| `matched_text_index` | The element is a text label the VLM matched to one of your `texts` entries | Index into the `texts` list you passed — look up the original `OcrResult` from `/infer` for its pixel-precise `bbox` |
| `bbox` | The element is a non-text UI component (icon, image, input) or no `texts` were provided | HTTP: `x, y, width, height` (top-left origin, pixels). SDK `LocateResult.bbox`: `x1, y1, x2, y2` pixel coordinates. |

The two are mutually exclusive. When `found` is `true`, exactly one of them is non-`null`.

<Tip>
  Grounded answers (via `matched_text_index`) are generally more accurate for text elements because the bounding box comes from the OCR pass rather than the VLM's own geometric reasoning. Always pass `texts` when your query targets a label or link.
</Tip>

## Model Selection

Omitting `model` (or passing `null`) uses the server's current default, which Axilio may update over time to reflect the best cost/accuracy trade-off. To pin a specific model — for reproducibility or cost control — pass its `id` from `GET /api/v1/inference/models`.

```python
# List available models and their pricing
import httpx

resp = httpx.get("https://api.axilio.ai/api/v1/inference/models")
for m in resp.json()["data"]:
    print(m["id"], "  input:", m["pricing"]["input"], "  output:", m["pricing"]["output"])
```

<Warning>
  If you pass a `model` ID that is not in the supported list, `/locate` returns a `400 Bad Request`. Call `GET /models` first to validate your choice programmatically rather than discovering unsupported models at runtime.
</Warning>

## When found Is False

A `false` result means the model determined the element described by `query` is not visible in the image — it is not an error condition. The VLM still runs and you are still billed for tokens. Build your automation to handle this case explicitly:

```python
location = client.argus.locate(image_bytes, query="the accept cookies banner")

if not location.found:
    # Banner may have already been dismissed — continue normally
    pass
else:
    # Tap the element — bbox uses x1,y1,x2,y2 pixel coordinates
    b = location.bbox
    driver.tap(x=(b.x1 + b.x2) // 2, y=(b.y1 + b.y2) // 2)
```

<Note>
  A low `confidence` score on a `found=True` result does not mean the element is absent — it means the model is uncertain about its exact position. Consider re-running with a more descriptive `query` or with additional OCR grounding if you see consistent low-confidence answers.
</Note>
