The Axilio Python SDK follows [Semantic Versioning](https://semver.org). Because the SDK is currently in early access (pre-1.0), the public API surface is still stabilising — minor version bumps may introduce breaking changes before `1.0.0` is released. Review the notes for each release before upgrading, and check [axilio.ai](https://axilio.ai) for the latest announcements, migration guides, and platform status updates.

<Info>
  **Early access — pre-1.0 SDK.** The `axilio` Python SDK has not yet reached `1.0.0`. Class names, method signatures, and module paths may change in any minor release. Pin to a specific version in production (`axilio==0.2.0`) and test upgrades in a staging environment before rolling them out.
</Info>

---

## How to Upgrade

Install the latest published version from PyPI at any time:

```bash
pip install --upgrade axilio
```

To upgrade to a specific version — recommended in production environments:

```bash
pip install "axilio==0.2.0"
```

After upgrading, verify your installed version:

```bash
python -c "import axilio; print(axilio.__version__)"
```

## Releases

### 0.2.0 — Current release

This is the first public early-access release of the Axilio Python SDK. It ships the full set of platform and driver APIs needed to build, run, and observe mobile automation workflows.

**What's included**

- **`Client`** — synchronous HTTP client with automatic retry and exponential backoff for transient errors (`RateLimitError`, `ServerError`). Context-manager support (`with Client() as client:`).
- **Platform resources** — `client.mobile`, `client.workflows`, `client.runs`, `client.api_keys`, `client.billing`, `client.usage`, `client.org`, `client.argus`.
- **Driver SDK** — `axilio.drivers.mobile` with the full on-device automation API: tap, swipe, type, wait, screenshot, and more.
- **Structured error hierarchy** — two independent exception hierarchies (`axilio` and `axilio.drivers.mobile`) with fine-grained error classes and a `.retryable` attribute on driver errors.
- **Mode detection** — automatic `Mode.LOCAL` / `Mode.SANDBOX` switching based on the execution environment, with `SandboxPermissionError` enforcing sandbox read-only constraints.
- **Configuration** — environment-variable-driven setup (`AXILIO_API_KEY`, `AXILIO_BASE_URL`, `AXILIO_SANDBOX_TOKEN`) with constructor overrides.

**Known limitations in this release**

- The async client is not yet available. All network I/O is synchronous. See [Upcoming Features](#upcoming-features) below.
- Streaming run events (real-time progress during a workflow run) are not yet exposed. See [Upcoming Features](#upcoming-features) below.
- The SDK requires Python ≥ 3.10.

## Upcoming Features

The following capabilities are actively in development. They are not available in `0.2.0` but will ship in upcoming releases.

<CardGroup cols={2}>
  <Card title="Async Client" icon="bolt">
    A fully `async`/`await`-compatible client for use with `asyncio`, `FastAPI`, and other async frameworks. The interface will mirror the synchronous `Client` exactly — you'll be able to migrate by swapping the import.
  </Card>
  <Card title="Streaming Run Events" icon="signal">
    Subscribe to real-time events emitted during a workflow run — step completions, device screenshots, log lines, and status transitions — without polling `client.runs.get()`.
  </Card>
</CardGroup>

## Upgrade Guide

<Accordion title="Upgrading within 0.x — what to check">
  Because the SDK is pre-1.0, any `0.x` → `0.y` bump may include breaking changes. Follow these steps before upgrading in production:

  <Steps>
    <Step title="Review this changelog">
      Read the release notes for every version between your current version and the target. Pay attention to any renamed classes, removed parameters, or changed default values.
    </Step>
    <Step title="Pin your dependency">
      Update your `requirements.txt` or `pyproject.toml` to pin the new version explicitly:
      ```
      axilio==0.2.0
      ```
    </Step>
    <Step title="Run your test suite">
      Run your automation tests against a staging environment before deploying to production. Focus on code that catches specific error classes — these are the most likely to change across minor releases.
    </Step>
    <Step title="Update error imports">
      If you import error classes by name, confirm the class names match the [Error Reference](/reference/errors) for the new version.
    </Step>
    <Step title="Deploy to production">
      Once tests pass in staging, deploy the upgraded SDK to production. Monitor error rates for the first few minutes after deployment.
    </Step>
  </Steps>
</Accordion>

<Accordion title="Checking your current installed version">
  Run either of the following commands to confirm which version you have installed:

  ```bash
  pip show axilio
  ```

  ```python
  import axilio
  print(axilio.__version__)
  ```
</Accordion>

## Versioning Policy

<CardGroup cols={2}>
  <Card title="Before 1.0.0" icon="triangle-exclamation">
    Minor version bumps (`0.x`) may include breaking API changes. Always review the changelog before upgrading. Pin to an exact version in any production deployment.
  </Card>
  <Card title="After 1.0.0" icon="shield-check">
    Once the SDK reaches `1.0.0`, it will follow strict Semantic Versioning: breaking changes only in major (`x.0.0`) releases, new backwards-compatible features in minor (`1.x.0`) releases, and bug fixes in patch (`1.0.x`) releases.
  </Card>
</CardGroup>
